Starting With Scala

Sep 24, 2019 by Colin Riddell

Scala is a functional programming langugage.

For years, I’ve always shied away from functional programming languages. I’ve done that becuase I always thought they were only really used by hard-core academic computer scientists to make themselves sound intelligent in front of others.

I’ve found out recently there are some really fun and powerful things to be learned from functional languages, such as Scala. They’re fast, good for asynchronous operations, have low latency and because of that they are becoming more prevelant and sought after to meet the real-time, low-latency demands of today.

ReactiveX, Observable and event-driven architectures are all important technologies and concepts we can’t ignore - and they borrow from functional programming.

I’m going to learn more about functional languages and share what strikes me as interesting about them, as I try to pick them up.

Scala is Fast

Scala runs on the Java Virtual Machine - before you get your pitchfork out, just, let me explain. Applications running on the JVM are not slow. The JVM merely has a slow startup time. Execution of code once the JVM has fired up is relatively fast.

It’s important to not mistake an applications startup time as meaning that the technology or programming language it was written in is slow. Compared to bare-metal, it of-course doesn’t compare, but studies do show Scala is faster than Go and Java. (Hundt, 2011 - link )

(DZone, 2012, https://dzone.com/articles/benchmarking-scala-against)

Scala is good for Asynchronous tasks

Well, Scala has promises. We’re familiar with promises from JavaScript. To recap, a promise is a placeholder object that’s given to the caller of a function or operation that’s not yet returned with the completed operation.

Think of a promise as being like a note with your order number on it that you get when you order a coffee in a caf√©. You’re very seldom handed your order immediately, so while it’s being made they give you a note. A note that promises that you’ll be given your coffee when it’s ready.

Promises and futures allow operations to happen asynchronously in a non-blocking way freeing the main thread to do something else. You see this on the web, a spinner loading icon will be rendered while the data is being fetched asynchronously. Without non-blocking asynchronous calls everything would freeze until it was loaded.

With Scala, since it’s working on the back-end the asynchronus stuff happens when doing some database transactions, or making further requests to other services.
